# 人机协作与日志追责规范

> 目的：统一 OrbitOS 中“用户写什么、Agent 写什么、自动化写什么”，并让所有 Agent 写入的日志都可追溯。

## 1. 适用范围

- 所有会写入 OrbitOS 的 Agent、Hook、自动化脚本、Dashboard 操作。
- 所有写入 `02-日记/` 的工作日志、反思、复盘、人际事件。
- 所有写入 `.orbit/events/*.ndjson` 的运行时事件。

## 2. 全局协作原则

### 2.1 用户拥有的内容

以下内容默认以用户为准，Agent 不得擅自代写或伪造：

- 主观感受、动机、价值判断。
- 对人际关系的解释、评价、立场。
- 未经用户确认的最终决策。
- 不确定真伪的事实细节、时间、责任归属。

### 2.2 Agent 可以代写的内容

以下内容允许 Agent 根据已有上下文进行结构化写入：

- 已完成工作的摘要。
- Git 提交、文件变更、工具执行记录。
- 已明确表达过的待办、下一步。
- 经用户确认后的决策摘要。
- 对已有材料的整理、归类、格式统一。

### 2.3 Agent 必须先确认的内容

以下内容在写入前应优先向用户确认；自动化场景无法确认时，只能写成“候选”或“待确认”：

- 对关系、人、策略的解释性结论。
- 对优先级、风险等级、责任归属的判断。
- 会影响后续执行方向的“明日计划”或“下一步”。
- 从日记区迁移到知识区、项目区、输出区的内容。

## 3. 日志追责原则

### 3.1 所有 Agent/自动化写入必须可追溯

凡是非纯手写内容，必须至少能回答：

1. 谁写的。
2. 什么时候写的。
3. 通过什么机制写的。
4. 来源上下文是什么。

### 3.2 标准追责字段

所有运行时事件记录必须包含以下字段：

- `event_id`：事件唯一 ID。
- `timestamp`：写入时间。
- `actor_type`：`user` / `agent` / `automation`。
- `actor_id`：写入方标识，如 `codex`、`claude-code`、`git-hook`。
- `recorded_via`：写入机制，如 `manual`、`dashboard`、`git-hook`、`stop-hook`、`record-agent-work-event`。
- `origin_cwd`：来源工作目录；无目录语义时仍应给出运行根。
- `session_id`：会话 ID；取不到时写 `unknown`。

## 4. 工作日志标准

### 4.1 Canonical sections

`02-日记/工作日志/YYYYMMDD_工作日志_周X.md` 的标准章节如下：

1. `## 今日重点`
2. `## 今日Todo`
3. `## Git 提交`
4. `## 重点记录`
5. `## 关键决策`
6. `## 问题与风险`
7. `## 明日计划`

不得再引入新的同级标准章节名作为默认模板。历史文件可保留，但新写入必须对齐上述结构。

### 4.2 section ownership

- `今日重点`：用户主导，Agent 可整理，不可凭空生成。
- `今日Todo`：用户确认后的待办为准；Agent 可提议和整理。
- `Git 提交`：自动化主导。
- `重点记录`：事实型过程记录；Agent 自动产出默认写这里。
- `关键决策`：仅记录已确认的决策。
- `问题与风险`：允许 Agent 提醒风险，但若属主观判断需标注“待确认”。
- `明日计划`：默认视为候选执行项，除非用户明确确认。

### 4.3 标准 Agent 记录格式

写入 `重点记录` 的 Agent 条目必须使用如下结构：

```markdown
### HH:MM — 标题

- 记录ID：`event_id`
- 记录方：`actor_id` (`actor_type`)
- 写入方式：`recorded_via`
- 会话ID：`session_id`
- 来源目录：`origin_cwd`

**为什么做**：...
**怎么做的**：...
**改了什么**：...
```

若无 `session_id`，写 `unknown`；不得省略 `记录方` 和 `写入方式`。

## 5. 多 Agent 接入约束

- 新 Agent 接入 OrbitOS 时，必须支持 `actor_id` 和 `session_id` 透传。
- 无法透传 `session_id` 的 Agent，也必须明确 `actor_id`，不能写成空值。
- 各 Agent 可以有自己的提示词和工作流，但不得自定义新的工作日志主结构。
- 各 Agent 若需写额外 metadata，应写在标准字段之外，不得替换标准字段含义。

## 6. todos 规则

- `02-日记` 中 Todo 的 canonical 位置是工作日志内的 `## 今日Todo`。
- `02-日记/todos/` 若存在，只能视为 Dashboard 或历史兼容层，不是事实源，不应成为新的默认写入目标。
- 其他 Agent 不应默认创建新的 `02-日记/todos/*.md`。

## 7. 迁移与兼容

- 历史工作日志中的 `Agent 产出`、`日志`、`待办事项` 等旧 section 可保留，不强制批量改写。
- 任何新日志、新自动写入、新模板，必须使用本规范。
- 若工具存在旧模板，应优先修工具，不靠人工记忆维持一致性。
