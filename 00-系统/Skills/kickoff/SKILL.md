# OBJECTIVE

将想法转为结构化项目。**先压力测试，通过后再立项**，避免在错误方向浪费精力。

## Step 0: 加载上下文

1. 读取 `.orbit/schema/subsystems.yaml`，确认 `04-项目` 工作区约束
2. 读取 `04-项目/WORKSPACE.md`，确认项目分类规则
3. 加载 `workspace-projects` Skill

## Step 1: 确定输入

1. 用户提供收件箱文件路径 → 读取内容
2. 用户提供内联描述 → 直接使用
3. 无输入 → 列出 `01-收件箱/` 中 `status: draft` 的文件

如果缺少关键信息（项目是什么、目标受众、期望结果），先追问再继续。

## Step 2: 压力测试

默认执行完整压力测试。用户可指定子模式：`问题验证` / `竞争分析` / `最小方案` / `深挖`。

**直接、具体、不讨好。像见过无数失败项目的评审，快速分离真实价值和一厢情愿。**

### 评分卡（6 维度，满分 30）

| 维度 | 得分 | 判断依据 |
|------|:----:|------|
| 价值清晰度 | /5 | 要解决什么问题？问题是真实的还是臆想的？ |
| 可行性 | /5 | 现在能做吗？依赖什么前提？技术/资源就绪？ |
| 紧迫性 | /5 | 为什么是现在？两个月后做有什么不同？ |
| 差异化 | /5 | 和现有方案有什么本质不同？"更好"不是差异化 |
| 验证速度 | /5 | 多久能拿到真实反馈？最小验证成本是什么？ |
| 匹配度 | /5 | 你在这个方向上有独特优势吗？还是纯靠努力？ |

分数必须基于具体证据，不能凭空打。"没有竞争"永远不对，当前做法就是竞争。

### 核心假设

一句话：什么必须成立，这个项目才值得做？这个假设必须在动手前可验证。

### 致命缺陷

| 风险 | 严重度 | 为什么致命 | 快速验证方式 |
|------|:------:|-----------|-------------|
| ... | 高/中 | ... | ... |

最多 3 个。按严重度排序。每个缺陷必须针对这个想法，不说"任何项目都有的问题"。

### 断言

- **通过**（≥ 18 分，无高危缺陷）→ Step 3 创建项目
- **调整**（12-17 分或有中危缺陷）→ 指出方向问题，用户决定
- **放弃**（< 12 分或有致命缺陷）→ 直接说为什么，建议归档

## 子模式（用户指定时触发）

### 问题验证

深挖问题真实性。角色：像客户调研专家，唯一验证方式是找到正在为此痛苦的人。

**步骤：**
1. 定义具体痛苦：什么时候发生？造成什么挫败？
2. 找出早期受众画像：谁最痛？不是泛泛的人群画像，是有具体行为的人
3. 写出 5 个发现性问题（问过去行为，不问假设意愿）：
   - ✅ "上次发生这个时你做了什么？"
   - ✅ "花了多少时间/钱？"
   - ❌ "你会用这个吗？"
4. 定义验证标准：什么信号证明问题是真实且紧迫的
5. 给出止痛药 vs 维生素判定

### 竞争分析

角色：竞争情报分析师。最危险的竞争对手往往是"现状"。

**步骤：**
1. 当前行为：目标受众现在怎么做？
2. 直接竞争：谁在解决同一个问题？
3. 间接竞争：谁在用不同方式解决同一个痛苦？
4. 真正的敌人：习惯？Excel？已有的工具？内部流程？
5. 差异化：凭什么让人切换？必须具体，不能只说"更好"

### 最小可行方案

角色：MVP 架构师。目的是用最小成本测试核心假设，不是做一个缩小版产品。

**步骤：**
1. 确认核心假设（从主测试中提取）
2. 定义最小功能集：只测假设的部分，其他全砍
3. 砍掉什么：列出不测核心假设的功能，明确不做什么
4. 定义验证标准：什么行为信号证明假设成立
5. 2 周计划：每天做什么，第 14 天前必须到真实受众面前

## Step 3: 创建项目（断言通过后）

1. **判断项目意图** → 一级分类：内容创作 / 产品系统 / 运营增长 / 研究验证 / 实验原型
2. **确定规模** → 单文件 / 目录项目
3. **创建文件**，使用 9 字段 Frontmatter：

```yaml
---
title: "项目名称"
type: roadmap
topic: project
workspace: "04-项目"
created: "YYYY-MM-DD HH:MM:SS"
modified: "YYYY-MM-DD HH:MM:SS"
tags: ["project", ...]
source: agent
status: active
project_type: content      # content|product|operations|research|experiment
project_category: "内容创作"
stage: planning
---
```

4. 项目首页必须包含"核心假设"和"首要风险"，来自压力测试结果

## Step 4: 清理收件箱

如果来自收件箱条目：`status: processed` → 移到 `99-归档/收件箱/YYYY/MM/`

## Step 5: 记录工作日志

在 `02-日记/工作日志/YYYYMMDD_工作日志_周X.md` 的 `## 重点记录` 追加。

## 输出格式

```
## 项目创建完成

**项目:** [[ProjectName]] 位于 04-项目/<分类>/
**压力测试:** 通过（X/30）
**核心假设:** ...
**首要风险:** ...
```
