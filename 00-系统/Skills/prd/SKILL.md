# PRD — 产品需求文档生成

## OBJECTIVE

将想法或压力测试结果转化为结构化的产品需求文档（PRD）。

**独立使用：** 用户直接说"写PRD" → 从对话中收集信息 → 生成PRD
**被kickoff调用：** 压力测试通过后 → 自动读取测试结果 → 生成PRD

## WORKFLOW

### Step 0: 收集上下文

**如果是kickoff调用：** 读取压力测试结果（评分卡、核心假设、致命缺陷），直接进入Step 1。

**如果是独立使用：** 通过苏格拉底式提问收集信息：

| 类别 | 关键问题 |
|------|---------|
| 问题定义 | 要解决什么问题？谁最痛？不解决的代价？ |
| 目标用户 | 主要用户是谁？有次要用户吗？ |
| 解决方案 | 打算怎么做？考虑过哪些替代方案？ |
| 成功指标 | 怎么衡量成功？北极星指标是什么？ |
| 范围 | 包含什么？明确不做什么？ |
| 约束 | 时间、技术、资源限制？ |
| 风险 | 最大的不确定性是什么？ |

一次只问2-3个问题，不要一次甩一堆。根据回答追问。

### Step 1: 匹配 Persona

根据项目领域，读取 `references/personas/` 下对应文件（与kickoff共享同一套persona）：

- 金融/交易 → Finance 系列
- 软件/技术 → SE 系列
- 通用决策 → General 系列
- 无匹配 → 跳过

### Step 2: 生成PRD

输出到 `04-项目/<分类>/<项目名称>.md`：

```yaml
---
title: "<项目名称>"
type: roadmap
topic: project
workspace: "04-项目"
created: "YYYY-MM-DD HH:MM:SS"
modified: "YYYY-MM-DD HH:MM:SS"
tags: ["project", "prd"]
source: agent
status: active
project_type: <content|product|operations|research|experiment>
project_category: "<分类>"
stage: planning
---
```

### Step 3: 确认

向用户展示PRD预览：

```
## PRD 预览

**项目:** <名称>
**分类:** <分类>
**核心假设:** <来自压力测试或对话>

### 文档结构
1. 问题陈述
2. 目标用户
3. 解决方案
4. 用户故事
5. 成功指标
6. 范围
7. 风险与依赖
8. 时间线

确认后保存到 04-项目/<分类>/<名称>.md
```

用户确认后写入。

### Step 4: 记录工作日志

在 `02-日记/工作日志/YYYYMMDD_工作日志_周X.md` 的 `## 重点记录` 追加。

## PRD 文档结构

```markdown
# <项目名称>

## 1. 问题陈述
[要解决什么问题，为什么现在解决]

## 2. 目标用户
[谁会用，用户画像]

## 3. 解决方案
[怎么做，核心功能描述]

## 4. 用户故事
作为 [用户类型]
我想要 [功能/行为]
以便 [价值/目的]

验收标准：
- [可测试的具体标准1]
- [可测试的具体标准2]

## 5. 成功指标
| 指标 | 目标值 | 衡量方式 |
|------|--------|---------|
| <北极星指标> | <目标> | <怎么测> |

## 6. 范围
### 包含
- [功能1]
- [功能2]

### 不包含（明确排除）
- [功能X — 为什么不做]
- [功能Y — 为什么不做]

## 7. 风险与依赖
| 风险 | 严重度 | 缓解方案 |
|------|--------|---------|
| <风险1> | 高/中/低 | <方案> |

## 8. 时间线
| 阶段 | 时间 | 产出 |
|------|------|------|
| MVP | 第1-2周 | 核心功能上线 |
| V1 | 第3-4周 | 完整功能 |
```

## 规则

- 先从压力测试结果（如有）提取信息，不足的部分再追问
- 用户故事用中文，不用 "As a..." 英文格式
- 成功指标必须可量化，不用"提升用户体验"这种模糊表述
- "不包含"部分必须明确写出，防止范围蔓延
- 所有文件使用 9 字段 Frontmatter
