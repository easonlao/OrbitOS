# 内容生命周期

这页说明一份材料进入 OrbitOS 后，如何变成可以长期使用的内容。

```text
原始输入 → 盘点建议 → 用户确认 → 已入库原件 → 项目 / 知识草稿 / 资源 / 输出 → 归档
```

## 1. 原始输入

文字、网址、图片、PDF、旧笔记和临时想法都可以直接进入 `01-收件箱/`。

[[../01-收件箱/00-粘贴|00-粘贴]] 是最低摩擦的文字入口，不要求 frontmatter。原始文件不需要为了格式整齐而改名或改写。

## 2. 盘点建议

Agent 第一次处理收件箱时只做：

- 识别大致主题。
- 判断与现有项目或知识是否相关。
- 提出去向建议并说明理由。
- 把需要决策的事项放到 [[../02-时间线/待确认|待确认]]。

此时不移动、不删除，也不直接生成正式知识。

## 3. 用户确认

你确认处理方向后，Agent 才能移动原件或创建长期内容。

确认的重点是“下一步怎么处理”，不要求你逐字审核所有机器记录。

## 4. 已入库原件

处理过的原始材料移动到 `01-收件箱/已入库/`，并留下机器记录。

“已入库”只表示原件已经处理过，避免 Agent 重复扫描；它不表示材料已经成为知识。原件仍保持完整，后续知识转写不能覆盖它。

## 5. 进入合适区域

### 项目

需要持续推进、包含任务或代码的内容进入 `03-项目/`。项目 `STATUS.md` 是当前项目状态源。

### 知识

值得长期复用的内容先转写到 `04-知识/00-草稿箱/`：

1. Agent 根据原件和证据形成可读综合。
2. 草稿明确来源、结论和不确定内容。
3. 你确认后，草稿才进入正式主题目录并变为当前知识。

[[06-术语表#当前知识|已确认知识]] 如果需要改变核心结论，必须先回到草稿箱，保留新旧证据和冲突说明，再由你重新确认。

### 资源

已知道用途、未来可能引用的图片、PDF、网页转存和附件进入 `05-资源/`。资源是证据或依赖，不等于知识结论。

### 输出

在 Obsidian 内形成的文章、报告和其他 Markdown 成品进入 `06-输出/`。项目代码和工程中间状态仍留在项目中。

## 6. 归档

归档表示退出当前工作视野，不是删除：

- 旧时间视图进入 `02-时间线/归档/`。
- 已结束或长期不活跃的完整对象进入 `99-归档/`。
- 原始证据、项目记录和知识不会因为归档而失去追溯关系。

## 哪些事情不会自动发生

OrbitOS 默认不会：

- 把收件箱内容直接变成知识。
- 覆盖已经入库的原始材料。
- 把未确认草稿当作当前知识。
- 静默移动或删除用户内容。
- 因为 [[06-术语表#Hindsight|Hindsight]] 记住了某件事，就认为它已经进入 OrbitOS。
