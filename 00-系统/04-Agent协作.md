# Agent 协作

OrbitOS 允许不同设备、平台和能力的 Agent 使用同一个工作台，但不会把它们假设成完全相同的执行者。

## Agent 身份

每个接入的 Agent 都有稳定 `agent_id` 和自己的 [[06-术语表#Agent Profile|Agent Profile]]，记录：

- 部署设备和局域网位置。
- OrbitOS 路径与接入方式。
- 可用运行环境和限制。
- 做过的重要事项。
- 自己的经验、踩坑和来源。

当前 Agent 入口见 [[agents/README|Agent 看板]]。

Agent 之间需要继续交接的事项，统一写到 [[agents/BOARD|Agent 交接板]]。具体 handoff 实例放在 `00-系统/agents/handoff/`，模板源放在 `.orbitos/templates/00-系统/agents/handoff/TEMPLATE.md`，执行流程见 `.orbitos/workflows/agent-handoff.md`。`BOARD` 只做索引摘要，不承载完整反馈链；真正的最新结论、本轮变化和下一步动作必须写在 handoff 顶部。

当 handoff 已经完成了方案提出、审核、异议处理和执行交接后，就说明协作流程本身已经收口。此时如果后续出现的新问题属于项目执行事实，例如实机验收失败、环境阻塞或新的配置缺口，这些内容应回到项目 `STATUS.md` 作为项目阻塞继续跟踪，而不是继续把 handoff 当作项目问题池。

## 第一次接入

未知 Agent 不会自动注册。它应先报告以下信息，由你确认后再注册：

- `agent_id`
- 部署位置
- 局域网 IP
- 接入方式
- OrbitOS 路径

注册完成后，Agent 才能执行写入型工作流。

## 每次开始工作

已注册 Agent 会先读取自己的轻量档案。这样 Hermes 的 Linux 限制、Nova 的平台行为和 Codex 的本地上下文不会混在一起，也不会要求你每次重新解释。

完整经验和历史问题不作为冷启动必读；只有任务命中相关领域、出现失败返工或需要排查历史问题时，Agent 才按档案中的经验入口按需展开。

读取完唯一入口和自己的轻量档案后，Agent 现在还应先判断这次任务属于哪种执行模式：

- `direct`：简单任务，单轮内可直接完成
- `delegated`：复杂任务，需要调查、审核、收口分阶段完成
- `confirm_first`：任务对象或能力边界仍不清楚，需要先问你

如果任务属于 `delegated`，Agent 不应再直接给一个单段最终答案，而应进入 handoff 协作层，留下角色分工、阶段完成证据、自信度与待确认边界。

除执行模式外，OrbitOS 还正式接入了一层轻量的 思考模块，系统设计定义见 `03-项目/OrbitOS/docs/DESIGN.md`。这层内容不是角色、技能或额外 workflow，而是让 Agent 在分析、拆解、表达和自检时，按问题特征自动推荐合适方法，并保持显式可见，且允许用户覆盖。

不同 Agent 可以处理共享项目，也可以只处理自己的事项。项目文件和 event 负责交接当前事实，轻量 Agent Profile 负责保留启动所需上下文，完整经验文件负责保留执行者特有经验。

当工作需要跨 Agent 继续时，先输出一份 handoff，再由审核把关后，由下一位 Agent 阅读最后确认后接手。审核结论、异议和下一步动作必须直接写回 handoff 或交接板，不依赖用户在不同 Agent 之间转述。为了避免接手方迷失，handoff 顶部必须有“最新结论”“本轮变化”“接手动作”三类摘要；研究型或未完全收口的交接，还必须显式写出置信度、信息状态和待确认项。

如果某次复杂交接的关键价值在于“本轮到底是怎么想清楚的”，handoff 可以只记录本轮采用的 思考模式和结论，不需要复制完整推理过程。思考模块在交接里的职责是帮助接手方理解判断方法，而不是把 handoff 变成新的长流程容器；模式既可以由用户显式指定，也可以由 Agent 自动推荐后显式标注。

当前 handoff 还要求显式写明 `交接类型`：

- `研究型`：重点交代证据、推断、待确认判断和二次核查，避免研究结论伪装成正式结果。
- `执行型`：重点交代已完成、未完成、接手动作和收口去向，避免执行链只剩背景说明而没有可继续动作。

研究型 handoff 默认更强调“未确认前要把待确认项摊开讲清楚”，但这不代表研究型永远不能收口。只要研究结论已经完成必要核查，并满足 `C3 + confirmed`，它就可以像执行型一样进入确认收口，不必再为了形式保留伪待确认状态。

这两类交接继续共用同一套 handoff 模板，不另建两套并行系统；分流主要通过字段、workflow 和 validation 实现。

一次完整的协作闭环至少包含四步：

1. A 根据用户需求写出方案或任务定义。
2. B 审核 A 的内容，并把结论直接写回 handoff。
3. A 根据审核结论完成收口，再把可执行版本交给执行方。
4. 执行完成后，如果只剩项目事实问题，就把阻塞写回项目状态，handoff 结束其协作职责。
5. 当 B 确认协作已经结束时，由 B 负责把 handoff 移入 `archive/`；A 不自归档自己的 handoff。

这类复杂协作任务还需要多一层“真实执行判定”：

- `automatic`：系统会按既有入口和 workflow 自然发生
- `manual_supported`：能执行，但必须由用户或 Agent 显式触发
- `concept_only`：当前只有设计或文档，没有真实执行路径

只有 `automatic` 能被描述为“系统能力已经成立”。如果还需要手动知道隐藏触发方式，或当前只是概念方案，就不能包装成自动能力。

## 经验如何积累

当对话中出现错误、返工、用户纠正、校验失败或明显有效做法时，Agent 先把可复用事实记录到自己的档案。

如果多个场景证明某条经验足够通用、原子化、可执行，它可以成为 learned rule；只有经过你确认，才会提升为所有 Agent 必须遵守的 [[06-术语表#Rule|core rule]]。

长篇、特定任务的方法通常更适合未来形成 Skill，而不是塞进规则。

## Hindsight 的位置

[[06-术语表#Hindsight|Hindsight]] 是可选的跨会话记忆增强层：

- Recall 可以帮助 Agent 找回长期事实和经验。
- Retain 只应保存稳定、可复用、经过确认的信息。
- Retain 之前必须先有本地主源，例如项目经验、确认知识、agent 经验或稳定环境事实。
- 当前默认适合 retain 的主要是稳定环境事实和长期经验；`project_status`、`intermediate_judgment`、`confirmed_knowledge`、`pending_confirmation`、完整 handoff 记录和临时草稿默认都不应直接进入 Hindsight。
- OrbitOS 文件、项目状态和 [[06-术语表#Event|Event]] 仍是最终可追溯依据。
- Hindsight 不可用时，OrbitOS 仍应正常运行。

## 定时 Agent

长期在线的 Agent 可以执行只读巡检和 validation watchdog。无人值守任务默认不移动文件、不创建知识、不修改规则；发现异常时报告，由你决定是否修复。
