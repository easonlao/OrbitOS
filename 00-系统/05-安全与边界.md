# 你的确认与内容安全

OrbitOS 的原则很简单：Agent 可以帮你推进工作，但不能替你决定内容的去留、结论的定稿或系统的扩大。

你不需要理解内部规则。只要知道什么时候应该看到确认，以及确认之后会发生什么。

## 这些事情一定会先问你

| 场景 | Agent 应怎样做 |
|---|---|
| 要移动、删除或归档你的材料 | 先说明会动哪些内容、去哪里，以及为什么 |
| 要把草稿变成正式知识、报告或其他成品 | 先给你看候选结果，等你确认再定稿 |
| 新旧知识相互矛盾 | 先说明冲突和可选处理方式，不擅自保留某一种结论 |
| 要新建项目、扩大到其他目录或改系统规则 | 先说明范围和影响，等你决定 |
| 要设置会写入内容的自动任务 | 先说明它何时运行、会改什么、异常时怎样处理 |

如果你只想让 Agent 先看看，不想发生任何变化，直接在请求后补一句：

```text
先给建议，不要修改或移动文件。
```

## 你随时可以要求更小的动作

下面几句话都可以用：

- `先盘点，不要处理。`
- `先写草稿，不要定稿。`
- `先告诉我影响，再改。`
- `只看这个项目，不要扩到其他目录。`
- `发现问题就报告，不要自动修复。`

短指令意味着 Agent 应缩小动作，而不是自行补全一个更大的任务。

## 系统怎样保护你的内容

- 收件箱是材料进入系统的唯一入口；Agent 不能跳过它直接把原始内容变成正式知识。
- 原始材料、项目当前状态、已确认知识与今日页各自承担不同职责，不会互相覆盖。
- 阅读笔记、人物档案和其他模块内容是否提升为长期结论，都需要经过相应确认。
- 工作完成后，系统会做结构和链接检查；如果检查失败，Agent 应报告问题，而不是把失败伪装成完成。

## 本地内容与产品更新

你的收件箱、项目、知识、时间线和个人协作偏好属于当前工作副本，不会因为更新 OrbitOS 产品文件而被当作公共内容上传或覆盖。

其中人物档案和本地协作偏好虽然放在 `00-系统/`，仍然是你的本地内容；它们不会自动变成其他用户的通用规则。

## 当你不确定时

直接问 Agent：

```text
这一步会修改哪些内容？哪些需要我确认？先不要执行。
```

这就是 OrbitOS 的默认协作边界：先看清影响，再让你拍板。
