# 安全与边界

OrbitOS 重点保护目录、状态源和需要用户决策的关键边界。

## Agent 可以直接做什么

- 读取与当前任务相关的最小上下文。
- 在明确范围内修改项目或系统文件。
- 更新项目状态和今日摘要。
- 运行 validation，发现结构或链接问题。
- 记录已经发生的操作事实和自己的执行经验。

## 哪些动作必须先确认

- 移动、删除或归档用户内容。
- 把草稿提升为当前知识或正式产物。
- 决定冲突说明草稿最终如何收敛为唯一当前知识。
- 新增或提升系统规则。
- 建立或替代重大架构决策记录（ADR）。
- 修改 Git 跟踪边界。
- 创建未知 Agent 的档案。
- 让无人值守任务写入 OrbitOS 内容。
- 把任务扩大到用户没有点名的项目或目录。

## 状态从哪里来

- `.orbitos/logs/events/` 通过 [[06-术语表#Event|Event]] 记录实际发生的操作事实。
- 项目 `STATUS.md` 记录项目当前状态。
- `04-知识/` 的 active 内容记录 [[06-术语表#当前知识|当前确认理解]]。
- `02-时间线/今日.md` 只是这些状态的 [[06-术语表#Dashboard|人读聚合]]，不反过来替代它们。

新证据与当前知识冲突时，OrbitOS 默认不会让两个冲突版本同时留在知识区；系统会先把两份提炼结果退回 `01-收件箱/已入库/`，再用一份冲突说明草稿等待你确认最终输出。

## Validation 的作用

写入型工作流完成后，Agent 会运行 [[06-术语表#Validation|Validation]] 检查：

- 必填字段是否缺失。
- 文件是否放在正确位置。
- 生命周期跳转是否合法。
- Obsidian 双链是否指向不该展示的机器文件。
- 入库记录与实际原件是否一致。

校验失败时应停止刷新 Dashboard，报告失败原因，而不是假装任务完成。

## Git 与本地内容

OrbitOS 系统文件通过 Git 更新；你的收件箱、时间线状态、Agent Profile、项目、知识、资源和输出默认不上传公共仓库。

开发 OrbitOS 本身时：

1. 在项目 [[06-术语表#Product Repo|Product Repo（repo/）]] 中修改并提交产品源码。
2. 推送到远端仓库。
3. 从 Product Repo 更新系统文件到 [[06-术语表#Runtime|Runtime]] 时，应使用适合当前本地部署形态的方式；更新不应覆盖本地用户内容和运行状态。

不要在 Runtime 根目录直接维护一套与产品仓库不同的系统文件。

当前 Runtime 专属的协作偏好是一个例外边界：它可以为了可读性放在 `00-系统/` 下，但语义上仍属于本地用户内容。像 [[08-本地协作偏好]] 这样的页面，默认不应进入 Product Repo；Product Repo 只保存它的通用模板与生成逻辑。只有确认其中的内容已经从个人偏好升级为通用约定后，才分别提升到系统说明或 Agent core rule。

## 核心原则

1. Obsidian 是人读界面，`.orbitos/` 是机器控制层。
2. Agent 开始前同步状态，完成后同步结果。
3. 只读取当前任务必要内容。
4. 原始证据、当前知识和状态视图彼此分开。
5. 长期内容需要可追溯来源，关键提升需要用户确认。
6. Hindsight 是可选增强，不是事实底座。
