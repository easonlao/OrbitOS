# OrbitOS

> 一个 Markdown 原生的人机协作工作台，用来协调人、agents、记忆系统和 Obsidian 可读产物。

OrbitOS 是一个 Obsidian-first 的人机协作工作台。它帮助你把零散输入、正在推进的事情和已确认的理解放进同一条可持续接力的工作节奏；[阅读](00-系统/模块/阅读.md)、[人物](00-系统/模块/人物.md)、[Hindsight](00-系统/模块/Hindsight.md)、[工程](00-系统/模块/工程.md)、[自动化](00-系统/模块/自动化.md)和[协作](00-系统/模块/协作.md)等能力只在你真正需要时再启用。

它的核心想法很简单：你在 Obsidian 里表达意图、阅读和确认；agent 负责整理、推进、记录与提醒。下次继续或换一位 agent 时，系统依然能告诉你做到哪里、为什么这样做、接下来可以做什么。

OrbitOS 现在还提供一套动态人物模块：模块本身属于 OrbitOS 的系统能力，但具体的 `00-系统/09-人物档案.md` 和本地协作偏好中的派生段落都只落在你当前使用的工作副本里，不会自动变成公共规则。首轮启用走一套 24 题、5 档倾向的基线问卷，MBTI 只作为低成本种子，不当作最终定论。

## OrbitOS 是什么

OrbitOS 不是传统个人知识库，也不只是 agent 框架说明书。它连接的是：

- Obsidian：人的阅读、确认和回看界面
- `AGENTS.md`：所有 agent 进入 OrbitOS 的统一入口
- `00-系统/`：用户需要理解的系统说明书
- `.orbitos/`：schema、日志、队列、workflow、rules、设计文档等内部运行层

## 最快开始

1. 下载或克隆这个仓库：

```powershell
git clone https://github.com/easonlao/OrbitOS.git
```

2. 在 Obsidian 里把克隆下来的 `OrbitOS` 文件夹打开为一个 vault。
3. 初始化当前工作副本：

```bash
python .orbitos/scripts/init-runtime.py
```

这个命令只创建缺失的本地文件，不会覆盖你已有的内容。
它会在当前工作副本里补齐 `01-收件箱/00-粘贴.md`、时间线、Agent registry 与模块状态表等核心运行文件，不会预先创建阅读、资源或输出目录。

4. 把你想丢进系统的东西复制到 `01-收件箱/`。
5. 如果要让 agent 接入，让它从 `AGENTS.md` 开始；具体执行规则都在 `AGENTS.md`，不是 README。

初始化后，你每天主要看 `02-时间线/今日.md`；它同时放需要你判断的事情和可以继续的入口。

## 如果你只想先用起来

你第一次其实只需要记住 4 件事：

1. 把材料先放进 `01-收件箱/00-粘贴.md` 或 `01-收件箱/`
2. 让 Agent 从 `AGENTS.md` 开始
3. 每天主要看 `02-时间线/今日.md`
4. 涉及移动、删除、归档、正式提升时，由你拍板

如果你不知道该对 Agent 说什么，可以直接这样开始：

```text
请先阅读 AGENTS.md 并执行 Startup Sync。
然后盘点一下收件箱，先给建议，不要移动文件。
```

## 日常怎么用

OrbitOS 的用户路径应该尽量短：

```text
把材料丢进 01-收件箱/
  -> 和 agent 聊天
  -> 看 `02-时间线/今日.md`
  -> 确认后的内容进入项目或知识
```

`01-收件箱/` 是临时入口：你可以先随手放文字、链接、截图说明或旧资料。agent 可以帮你盘点、摘要、分组，并建议放到哪里；但不能未经你确认就把这些内容变成正式知识卡片、项目产物或发布内容。

最常见的几种用法可以直接这样说：

- `整理一下收件箱，先给建议，不要移动。`
- `继续推进某个项目，先告诉我当前做到哪里。`
- `把这份已入库材料整理成知识草稿，先不要定稿。`
- `请启用动态人物模块，走人物基线初始化流程。`
- `同步进度。`

## 仓库结构

```text
AGENTS.md              # agent 进入系统时必须读取的规则入口
README.md              # 中文上手说明
00-系统/               # 给用户看的系统说明书
01-收件箱/             # 临时入口，先放原始材料
02-时间线/             # 今天和本周等时间视图
03-项目/               # 每个项目自己的资料和状态
04-知识/               # 已确认、值得长期保留的知识
.orbitos/              # agent 和脚本使用的内部文件，普通使用时不用打开
```

阅读等可选能力启用后才会出现自己的目录。例如阅读模块启用后创建 `05-阅读/`；`06-资源/` 与 `07-输出/` 仍是保留编号，目前没有可安装模块。

## 可选模块

先查看当前状态：

```bash
python .orbitos/scripts/module-manager.py status
```

安装与启用某个模块是两个步骤。以阅读为例：

```bash
python .orbitos/scripts/module-manager.py install reading
python .orbitos/scripts/module-manager.py enable reading
```

`enable` 只创建该模块声明的空结构，不会移动或改写你的内容。人物、Hindsight、工程模块还需要完成各自配置后才能进入 `ready`；工程模块还需要单独安装 Matt Pocock Skills。

## Agent 协作

README 只说明你作为用户怎么使用 OrbitOS。agent 的唯一执行入口是 `AGENTS.md`。

你只需要知道四件事：

- 新 agent 第一次接入时，应该先读 `AGENTS.md`。
- 未注册的 agent 需要先向你确认身份和部署信息，不能直接写入系统。
- 工作完成后，你可以说"同步""同步进度"或"更新进度"，让 agent 按 `AGENTS.md` 刷新时间线和状态。
- 如果当前工作副本已经有 `09-人物档案.md` 或本地协作偏好，agent 只会按命中任务按需读取；这些内容属于当前工作副本的本地实例，不等于所有用户通用的系统规则。

## 修改 OrbitOS 本身

日常使用不需要打开 `.orbitos/`。

只有当你要修改 OrbitOS 的规则、workflow、schema、目录协议或发布流程时，才需要查看命中的实现文件、测试与验证脚本。产品不再提供额外的 `.orbitos/` 开发者入口。

运行最小验证集：

```bash
python .orbitos/scripts/run-validation.py
```

如果 Python 不可用，可使用 Node.js fallback：

```bash
node .orbitos/scripts/run-validation.mjs
```
