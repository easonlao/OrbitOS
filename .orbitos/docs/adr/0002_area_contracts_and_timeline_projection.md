# ADR-0002：按区域定义数据契约，时间线只保留投影

- 状态：accepted
- 日期：2026-06-20
- 范围：OrbitOS 数据流、规则路由与时间线架构

## 上下文

收件箱、时间线、项目、知识和资源处理的数据不同，但现有规则主要集中在 core，无法清楚回答每个区域接收什么、以什么为权威来源以及可以流向哪里。同时，`待确认.md` 与 `下一步.md` 聚合多个区域的信息，容易与项目 STATUS、ROADMAP 和其他区域状态重复并逐渐陈旧。

只按文件名补充职责不能解决问题。OrbitOS 需要先确定数据从哪里产生、在哪个区域成为权威状态、如何经过 workflow 流转，以及何时投影给用户。

## 决策

采用三层控制结构：

- `.orbitos/rules/core/` 保存跨区域共同约束。
- `.orbitos/rules/areas/` 分别保存各可见区域的数据契约。
- `.orbitos/workflows/` 实现区域内部与跨区域的具体流转。

根 `AGENTS.md` 只根据任务对象路由最小区域规则。可见区域不默认增加 `AGENTS.md`；只有目录树存在必须逐级继承的局部约束时才创建。

时间线只保留 `今日.md` 和 `本周.md` 等时间投影。待确认事项保留在产生它的区域状态中，当前任务与长期方向分别保留在项目 STATUS 和 ROADMAP；当天需要用户看到时，由 `今日.md` 提供摘要和来源入口。投影不得反向覆盖权威来源。

## 备选方案

- 所有区域共用一份大规则：入口较少，但区域边界会继续混杂，Agent 需要读取与任务无关的内容。
- 每个可见目录都创建 `AGENTS.md`：局部入口直观，但会增加默认读取、重复共同规则并提高维护成本。
- 保留独立 `待确认.md` 和 `下一步.md`：跨项目浏览方便，但会形成额外状态源，需要持续同步且已经出现陈旧内容。

## 后果

- 每个区域的数据入口、权威来源和出口可以独立维护与验证。
- 跨区域共同约束只维护一份，区域切换时按语义加载最小规则。
- Progress Sync 必须先更新权威来源，再写 Event，最后按需刷新 `今日.md`。
- `待确认.md` 与 `下一步.md` 需要在内容迁移后删除；相关设计、规则、workflow、初始化模板和自动检查必须同步调整。
- 新增区域时必须先定义区域契约；只有存在真实流转时才新增 workflow。
