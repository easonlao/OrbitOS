# Reading Module — 领域模型

本文档定义阅读模块的术语、对象关系、状态机和关键决策。实现细节在各 SKILL.md 和 workflow 中；本文档只回答"是什么"和"边界在哪"。

## 术语表

| 术语 | 定义 |
|------|------|
| **书 (Book)** | 一次深读的完整对象，包含有序章节、来源和进度。 |
| **章 (Chapter)** | 书的一级内容单元，包含原文底本和一个或多个阅读单元。 |
| **阅读单元 (Reading Unit)** | 章内一次注意力可承受、语义自足的完整阅读片段。 |
| **原文底本 (raw.md)** | 章节原文的只读源头，任何流程不得改写或删除。 |
| **9 段笔记** | 每个阅读单元的标准化理解框架，见下方结构 | `chNN/NN.md` 正文 |
| **批注 (Annotation)** | 贴在单元原文某段上的划线笔记，用于保留具体文本位置上的重读发现。 |
| **洞察 (Insight)** | 阅读现场的个人沉淀，不自动进入知识库或 Hindsight。 |
| **阅读进度 (Progress)** | 记录章节和阅读单元顺序，以及各自的真实完成状态。 |
| **章节提炼 (Chapter Distillation)** | 章级深读完成后的总结性提炼，覆盖全章所有单元的理解 | 独立于单元笔记 |
| **深读 (Deep Reading)** | 围绕单元原文的交互式理解过程：展示→反馈→讨论→记录 | 对话 + 单元笔记 |
| **冷读审查 (Cold Read Review)** | 章节切分后由独立 Agent 做的切分质量评审。 |

## 9 段笔记结构

每个阅读单元 `NN.md` 包含以下 9 段：

| 段号 | 标题 | 生成时机 | 内容性质 |
|------|------|----------|----------|
| 一 | 原文 | 切分时写入 | 单元原文（从 raw.md 切出） |
| 二 | 字词注 | 深读时渐进生成 | 关键字逐个解 |
| 三 | 白话直译 | 深读时渐进生成 | 尽量贴字面 |
| 四 | 注家对照 | 深读时按需生成 | 有可靠对照材料时区分来源与观点 |
| 五 | 与上一单元/前章的连接 | 深读时渐进生成 | 上下文衔接 |
| 六 | 三个值得停下来想一想的问题 | 深读时渐进生成 | 启发性提问 |
| 七 | 与生活的关联 | 深读时渐进生成 | 把抽象拉回经验 |
| 八 | 你的理解/疑问 | 用户表达并确认后记录 | 用户观点，不由 Agent 发明 |
| 九 | 回看批注 | 用户明确要求回看时记录 | 默认留白 |

**生成规则**：第 1 段在切分时写入；第 2—7 段在用户反馈后于深读过程中渐进生成；第 8/9 段保持用户主权，只有符合记录条件时才落盘。

## 对象关系

```text
书 (Book)
 └── progress.md（章→单元两层）
 └── .orbitos-source.json（来源追踪）
 └── chNN/（章）
      ├── raw.md（只读原文底本）
      ├── 00-导读.md（长章切分后的卷层入口，可选）
      ├── NN.md（阅读单元，含 9 段笔记）
      └── NN.notes.md（批注旁车，可选）
```

- 一本书有 1..N 章
- 一章有 1..N 个阅读单元（短章 = 1 个单元，长章切分为多个）
- 每个单元有 1 份 9 段笔记（`NN.md`）
- 每个单元可有 0..N 条批注（`NN.notes.md`）
- 章级提炼不在单元文件中，是深读完成后的独立产物

## 状态机

### 单元状态

```text
骨架 (skeleton)
  ↓ 用户开始读
已展示 (shown)
  ↓ 用户反馈
用户已反馈 (user-responded)
  ↓ Agent 展开讨论
讨论中 (discussing)
  ↓ 讨论结论写入笔记
已记录 (recorded)
  ↓ 所有 2-7 段填充完毕
单元完成 (unit-complete)
```

### 章状态

```text
未切分 (unsplitted)
  ↓ chapter-split 执行
已切分 (split) — 所有单元为骨架态
  ↓ 逐单元深读
部分深读 (partially-read) — 部分单元完成
  ↓ 所有单元深读完成
所有单元完成 (all-units-complete) — ≠ 章节深读完成
  ↓ 章节提炼完成
章节提炼完成 (chapter-distilled)
  ↓ 章节深读审查通过
章节深读完成 (chapter-deep-read)
```

**关键语义区分**：
- **单元完成** = 该单元的 2-7 段在深读中渐进填充完毕
- **章节提炼完成** = 所有单元完成 + 章级总结性提炼已生成
- **章节深读完成** = 章节提炼完成 + 独立审查确认理解无重大遗漏

**覆盖所有单元 ≠ 章节深读完成**。必须显式完成章节提炼和审查才能标记章节深读完成。

## 关键决策

1. **raw.md 只读**：任何流程不得改写、删除或覆盖 `raw.md`。它是切分和溯源的唯一底本。
2. **2-7 段渐进生成**：第 2-7 段在深读过程中逐步填充，不在切分时预生成。
3. **8/9 段用户主权**：第 8 段只记录用户确认过的理解/疑问，第 9 段只在用户明确要求回看时写入；Agent 不替用户发明观点。
4. **切分依据是理解**：切分点由内容理解决定，不是字数。三层标准：注意力、语义自足、信息密度。
5. **冷读审查必须独立**：章节切分后必须由独立子 agent 做冷读评审，不可自评。
6. **覆盖完单元 ≠ 章节完成**：必须完成章节提炼和深读审查才能标记章节深读完成。
7. **批注只在 NN.md 上**：洞察、INDEX 等衍生文件不写批注旁车。
8. **对话内容不自动持久化**：讨论中的临时交流只在对话上下文中；只有确认过的结论、修正、未解问题或用户要求的批注才进入阅读文件。

## 模块边界

- 阅读数据留在 `05-阅读/`；`books/`、`progress.md`、批注和洞察不成为项目状态、知识或 Hindsight。
- 不在阅读域放 Agent 专属 hooks、settings、插件或 dashboard 配置。
- 模块实现位于 `.orbitos/modules/reading/`；用户数据位于 `05-阅读/`。
