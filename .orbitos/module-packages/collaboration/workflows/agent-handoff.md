# Agent Handoff Workflow

Agent Handoff 用来承接 `execution_mode=delegated` 的复杂协作任务。它只负责协作交接，不替代项目 `STATUS.md`、知识卡片或 event。

## 目标

- 让交接对象、交接边界和下一步动作显式化。
- 让复杂任务的执行模式、角色分工和阶段完成证据显式化。
- 让中间判断不会伪装成正式结论。
- 让接手方异议和局部驳回可以被结构化处理。
- 让 handoff 能进入 validation，而不是停留在自由文本。
- 让“这项能力是否真的会自动发生”与“这只是手动支持或概念方案”可以被区分。

## 触发条件

出现以下任一情况时，应创建或更新 handoff：

- 当前任务已判定为 `execution_mode=delegated`。
- 用户明确说“交给另一位 Agent 继续”“交接当前任务”或“把任务交给 {agent_id}”。
- 当前任务需要由另一位 Agent 或另一轮会话继续。
- 当前结论已经形成，但仍处于研究收口、审核、执行交接或异议处理阶段。
- 项目目录适合继续记录“项目事实”，但不适合承接多 Agent 的细粒度协作链。
- 接手方需要直接看到边界、证据、风险和下一步，而不是依赖用户转述。

以下情况通常不需要 handoff：

- `execution_mode=direct` 且纯单 Agent、单轮内即可收口的工作。
- 只剩项目执行事实或项目阻塞，且不再需要协作审核链。
- 只是普通状态同步，没有实际交接对象。

## 最小输入

创建或更新 handoff 前，至少先确认：

- 交接标题和项目归属。
- `execution_mode` 与本轮角色分工。
- 当前任务目标、本轮边界和明确不处理项。
- 最新结论、本轮变化和接手动作。
- 置信度、信息状态和是否需要二次核查。
- 关键证据入口、已确认判断、待确认判断。
- 是否触发驳回；若触发，回退范围是什么。
- 当前交付到底属于 `automatic`、`manual_supported` 还是 `concept_only`。

## 执行流程

1. 判断当前事项是否属于“协作交接”，而不是项目状态更新或知识沉淀。
2. 在 `00-系统/agents/handoff/` 新建或更新 handoff，使用 `.orbitos/templates/00-系统/agents/handoff/TEMPLATE.md`。
   - 用户明确指定接手 Agent 时，`交给谁` 写入其 `agent_id`。
   - 用户未指定接手 Agent 时，`交给谁` 写入 `待接手`；接手方可通过“获取交接工作”定位它。
3. 先写顶部摘要：
   - `最新结论`
   - `本轮变化`
   - `接手动作`
4. 先补齐项目归属：
   - `项目`
   - `路径`
   - `是否为项目内任务`
   如果这三项任何一项还说不清，先暂停并确认，不要继续把 handoff 写成可接手状态。
5. 再补齐任务边界、当前阶段、证据入口和风险阻塞。
   - 在 `任务边界` 中显式写明 `执行模式`；handoff 只接受 `delegated`。
   - 在 `任务边界` 中显式写明 `交接类型`：`研究型` 或 `执行型`。
   - `研究型` 重点交代证据、推断、待确认点和二次核查。
   - `执行型` 重点交代已完成、未完成、接手动作和收口去向。
   - 如本轮结论明显依赖特定思考方法，可补一行 `本轮思考模式`，只写模式名与用途，例如 `5W1H: 补齐需求维度`、`反向推导: 倒推出第一步`；模式可以来自用户显式指定，或由 Agent 自动推荐后显式标注，不要求复制完整推理过程。
   - 只有当本轮实际采用了思考模式时，才写 `本轮思考模式`；未采用时不写，不补“未使用”说明。
   - 在 `本轮角色分工` 中写清：
     - `roles_required`
     - `role_assignments`
     - `completion_contract`
   - 在 `当前阶段` 中写清：
     - `阶段`
     - `阶段完成证据`
     - `是否已满足交付合同`
6. 按当前 handoff 模板与 validation 的字段约束填写：
   - `置信度`
   - `当前信息状态`
   - `需要二次核查`
   - `已确认判断`
   - `待确认判断`
   - 如有 `本轮思考模式`，只把它当作判断形成方式的摘要，不把它写成新的强制字段或新 workflow，也不要求每次交接都必须出现。
7. 在 `真实执行判定` 中写清：
   - `execution_reality`
   - `触发路径`
   - `缺失原因`
   其中：
   - `automatic`：系统无需额外隐藏触发方式就会自然发生
   - `manual_supported`：能执行，但必须由用户或 agent 显式触发
   - `concept_only`：当前只有设计或文档，没有真实执行路径
8. 当 handoff 涉及“这条内容该写到哪里”时，再读取 `03-项目/OrbitOS/docs/INFORMATION-ROUTING-PROTOCOL.md`，确认它属于项目状态、协作记录、中间判断、长期经验还是临时草稿。
9. 接手方如仅补细节或提出优化建议，写入 `接手方异议`；如发现关键错误或无法继续依赖，转入 `驳回与回退`。
10. 当协作链已经完成，后续只剩项目执行事实或阻塞时，把事实转回项目 `STATUS.md`，不要继续把 handoff 当项目问题池。
11. 在 `00-系统/agents/BOARD.md` 的“当前进行中的 handoff”登记该 handoff 的可点击链接、标题与简短状态，确保接手 Agent 可从交接板定位它。
12. 在 Progress Sync 前运行 `python .orbitos/scripts/run-validation.py`，确认 handoff 结构、项目归属、角色执行证据和真实执行判定都通过检查。

## 驳回边界

以下情况进入普通异议，不算驳回：

- 补充信息。
- 调整表述。
- 给出更优建议，但原路径仍可继续执行。

以下情况进入驳回：

- 关键事实错误。
- 关键前提缺失。
- 低置信度或待确认内容被误当成正式依据。
- 下游已无法按原交付继续推进。

只要触发驳回，就必须补齐：

- `是否触发驳回`
- `驳回范围`
- `仍然有效的部分`
- `必须回退的部分`

## 与其他状态源的分工

- `handoff`：承接协作链、判断边界和接手动作。
- `handoff`：承接复杂任务的执行模式、角色合同、阶段证据、判断边界和接手动作。
- 项目 `STATUS.md`：承接项目现状、阻塞和后续项目事实。
- event：记录本次实际操作。
- Hindsight：只在需要跨会话长期复用且已确认稳定时 retain。

## 执行清单

### 进入检查

- [ ] 当前事项确实需要跨 Agent 或跨会话继续。
- [ ] 已判定本轮 `execution_mode=delegated`，而不是 `direct` 或 `confirm_first`。
- [ ] 已确认这不是单纯的项目状态更新。
- [ ] 已确认交接对象和接手目标。
- [ ] 已确认项目归属、路径和是否为项目内任务，不存在待定项。

### 执行检查

- [ ] 已使用 handoff 模板，而不是自由发挥正文结构。
- [ ] 已写明项目归属、执行模式、任务边界、交接类型、最新结论、本轮变化和接手动作。
- [ ] 已填写角色分工、阶段完成证据和交付合同状态。
- [ ] 已填写置信度、信息状态和二次核查要求。
- [ ] 已区分已确认判断与待确认判断。
- [ ] 已填写真实执行判定，未把手动流程或概念方案误写成自动能力。
- [ ] 触发驳回时已写清局部回退范围。

### 退出检查

- [ ] handoff 已通过 validation。
- [ ] 后续只剩项目事实时，已转回 `STATUS.md` 而不是继续堆在 handoff。
- [ ] Progress Sync 已记录本轮交接更新。

## 禁止

- 不把 handoff 当作项目长期问题池。
- 不把 `execution_mode=direct` 的简单任务强行包装成 handoff。
- 不把低置信度或待确认内容伪装成“已确认判断”。
- 不把接手方关键驳回写成零散评论。
- 不在 handoff 中复制 Agent 内部 `soul / user / memory / skill / config` 详情。
- 不把 E2 思考方法层写成一套必须逐步执行的固定流程。
