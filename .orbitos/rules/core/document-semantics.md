# Document Semantics Rule

本规则统一 OrbitOS 固定角色 Markdown 的受众和职责。创建、修改或审查这些文件时必须遵守。

## 核心原则

- 一个文件只回答一类问题，不通过复制内容制造多个状态源。
- 人读入口与 Agent 入口分离；用户说明不能成为 Agent 的隐式执行规则。
- 当前状态、执行队列、阶段路线、未确认决策和历史理由互不替代。

## Markdown 创建门

创建新的可见 Markdown 前，Agent 必须先向用户说明：

1. 新文件要解决的问题和唯一职责。
2. 现有文件为什么不能承载这些内容。
3. 计划使用的路径、受众和生命周期。

用户已明确要求创建指定文件时，该请求视为确认；否则必须等待用户确认后再创建。不能因为一种内容有常见文件名，就默认增加文档。

## 固定角色

| 文件 | 主要受众 | 只回答 | 不应包含 |
|---|---|---|---|
| `MAP.md` | 用户 | 当前区域是什么、有哪些直属子目录入口 | 下层文件导航、原则、规则、状态、目录协议、Agent 指令 |
| `README.md` | 产品用户 | 可独立发布或使用的对象是什么、如何安装或使用 | 内部项目管理、Agent 必读规则、当前状态或历史 |
| `AGENTS.md` | Agent | 当前对象的目标、稳定架构、执行规则、读取路由和停止条件 | 用户教程、内容地图、当前进度、下层规则全文 |
| `STATUS.md` | 用户与 Agent | 对象现在处于什么状态、当前最多 3 项任务和 3 项待确认是什么 | 完整历史、远期路线、任务池或问题池 |
| `ROADMAP.md` | 用户与 Agent | 目标、总体状态和完成条件清单 | 当前任务、每日进度、完整操作历史 |
| `CHANGELOG.md` | 开发者与维护者 | 已落地的重要版本变化、迁移点和历史摘要 | 当前任务、每日进度、完整 event 流水、未落地设计讨论 |
| ADR | 开发者与 Agent | 为什么做出重大且难回退的选择 | 当前状态镜像、任务进度、普通变更流水 |

## MAP

- 可以用一句话说明当前区域是什么。
- 只导航当前区域的直属子目录；每个子目录链接一个用户入口，并用一句话说明它是什么。
- 不下钻列出子目录内部的 STATUS、ROADMAP、docs 或具体内容；进入子目录后由它自己的入口继续导航。
- 没有足够稳定入口时不新建 MAP；发布模板中的空 MAP 只能作为明确占位。
- 状态进入 `STATUS.md`，约束进入规则或 `AGENTS.md`，稳定概念说明进入产品 README 或用户文档。

## README 与 AGENTS

- README 不再是每个项目或目录的默认文件。只有发布仓库、可独立使用的模块或明确的用户入口需要 README；内部项目管理目录默认不创建。
- README 可以告诉用户 `AGENTS.md` 在哪里，但不能保存 Agent 执行所必需的内容。
- Agent 不默认读取 README；只有任务涉及 README、用户介绍或 onboarding 时才读取。
- 根 `AGENTS.md` 管全局生命周期和硬边界；项目级 `AGENTS.md` 还应让 Agent 无需读取 README 就能理解项目目标、稳定架构和项目规则。
- 更深层的 `AGENTS.md` 只补充其目录树内必要的目标上下文、路由与约束。
- 局部 `AGENTS.md` 不重复 Startup Sync、全局规则或更深层入口的正文。

## 状态类文档

- `STATUS.md` 是当前镜像，并直接保存当前任务；完成历史进入 event、changelog 或 ADR。
- `STATUS.md` 保存详细现状和判断依据；属于长期路线的当前任务标注 ROADMAP 编号，其他跨会话小任务标注“临时事项”，不得强行提升为路线目标。
- 同一项目默认最多保留 3 项当前任务。加入第 4 项前，必须先完成、移除或明确替换已有任务，不通过增加文档扩张执行队列。
- 待确认决策默认直接写入 `STATUS.md`，最多 3 项；确认后立即移除并更新正式来源。只有出现不同且稳定的职责时，才能按 Markdown 创建门提议独立文档。
- `ROADMAP.md` 保存目标、总体状态和完成条件清单，但不展开当前任务或完整过程历史。
- 完成条件按实现逻辑排列；已验证项标记 `[x]` 和 event 可证明的日期，未完成项标记 `[ ]`。完整时间线仍留在 event。
- 目标状态必须来自 STATUS；只有完成条件、validation 和 event 证据同时满足时，才能在同一次 Progress Sync 中更新 STATUS 与 ROADMAP。
- `CHANGELOG.md` 记录已经落地并值得进入历史的变化，可按版本或阶段批次维护，但不承担当前项目进度或里程碑状态。

## 审查清单

1. 这份文件的主要受众是谁？
2. 内容是否回答了该文件唯一负责的问题？
3. 是否复制了另一个状态源、规则或入口的正文？
4. Agent 是否被迫读取 README 或 MAP 才能行动？
5. 用户是否被迫阅读 AGENTS 或内部规则才理解当前状态？

任一答案暴露职责混用时，先把内容移回权威文件，再保留最短链接或路由。
