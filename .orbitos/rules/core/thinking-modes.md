# Thinking Modes Reference

本文件是 OrbitOS 思考方法层的运行时详细词典。Agent 在 `.orbitos/rules/core/thinking.md` 选定某种方法后，只读取该方法对应章节，用它指导实际分析、拆解、表达或复核。

它不是角色、Skill 或额外 workflow，也不要求一次加载或执行全部八种方法。

## 1. 总体判断

这些模式不是角色，不是技能，更不是新的流程系统。

- 角色回答“这次谁负责什么职责”。
- 技能回答“这个 Agent 能调用什么能力”。
- 思考模式回答“当前该用什么脑回路处理问题”。

因此，思考方法层是 OrbitOS 的认知层：在不同任务、不同阶段和不同不确定性下，为 Agent 提供可切换的思考方法。

## 2. 接入原则

E2 已正式接入 OrbitOS，但接入方式保持克制：

- 不把 8 个模式写进每个 Agent 的内部画像，不复制 `soul / user / memory / skill / config`。
- 默认采用自动推荐、显式可见、用户可覆盖的调用方式。
- 不要求每次任务都显式跑完整套模式，只在当前问题需要时按需调用。
- 不把思考方法层变成固定流程图；它只约束思考质量，不增加额外执行步骤。
- 允许多个模式组合使用，但组合顺序由任务决定，不做硬编码编排。
- 详细定义集中在本页；`AGENTS.md` 只保留短入口，不重复正文。

## 2.1 调用口径

当前统一口径如下：

- 用户已显式指定模式时，以用户指定为准。
- 用户未指定时，Agent 可以自动推荐 1 到 2 个最合适的模式。
- 推荐后应保持显式可见，例如在分析摘要、handoff 或结论前标注 `本轮思考模式`。
- 用户可以覆盖 Agent 的推荐，不需要遵循默认推荐。
- 简单任务可以不触发任何模式。

## 3. 系统接入面

当前 E2 在 OrbitOS 中的正式接入面只有 3 个：

### 3.1 分析阶段

当任务需要澄清、拆解、重构或判断时，Agent 可以在内部分析时主动推荐合适模式；如果已经采用，应尽量保持结果可见。

### 3.2 handoff 阶段

当复杂任务进入 `execution_mode=delegated` 且需要交接时，handoff 可以显式写明本轮采用了哪些思考模式，用来解释当前结论是如何形成的。

### 3.3 review / 自检阶段

当方案、判断或结论准备提交时，Agent 可以用 `金字塔原理`、`苏格拉底提问`、`六顶思考帽` 或 `批判性思维` 做最终整理与补强。

这 3 个接入面只负责“何时可以引用 E2”，不把 E2 写成强制结构化话术，也不要求每次都先问用户该选哪个模式。

## 4. 模式清单

当前收敛出的模式共 8 个：

1. `5W1H`
2. `苏格拉底提问`
3. `SWOT 分析`
4. `第一性原理`
5. `反向推导`
6. `金字塔原理`
7. `六顶思考帽`
8. `批判性思维`

## 5. 模式分组

为便于后续路由，先按主要作用分成 4 组：

### 3.1 澄清类

- `5W1H`
- `苏格拉底提问`

### 3.2 判断与重构类

- `SWOT 分析`
- `第一性原理`

### 3.3 拆解与推进类

- `反向推导`

### 3.4 表达与校验类

- `金字塔原理`
- `六顶思考帽`
- `批判性思维`

这只是当前的主用途分组，不代表每个模式只能出现在一个阶段。

## 6. 单模式定义

### 4.1 5W1H

#### 核心定义

在多场景系统中，通过 `What / Why / Who / When / Where / How` 六个维度扫描对象，保证每个维度都有明确输出，从模糊走向全覆盖、无盲区。

#### 解决的问题

- 任务理解不完整。
- 方案覆盖不全。
- 有些关键维度没有被显式回答。
- 看起来说了很多，但仍然存在结构性遗漏。

#### 适用场景

- 任务进入系统时的 intake。
- 项目定义。
- 需求澄清。
- handoff 信息补全。
- 方案盘点。

#### 触发信号

- 任务目标还模糊。
- 觉得信息零散，但说不清缺了什么。
- 担心某些维度没看见。

#### 典型输出

- 六维扫描结果。
- 每个维度对应的已知项、未知项、待确认项。
- 当前任务的全景定义。

#### 不适用场景

- 已经进入深度论证和推理校验阶段。
- 只差最后表达整理的时候。

#### 一句话

`5W1H` 是横向补全，用来防遗漏。

### 4.2 苏格拉底提问

#### 核心定义

当一个结论表面上看没问题、但还不够扎实时，通过分层追问假设、证据、替代视角与后果，把隐藏的逻辑缺口逼出来，让结论经得起追问。

#### 解决的问题

- 结论看似成立，但逻辑链不够硬。
- 假设没有被说出来。
- 证据和结论之间存在跳步。
- 没想过其他解释。

#### 适用场景

- 方案审查。
- 结论复核。
- 决策前质询。
- handoff 后二次追问。
- 知识转写时加固论证。

#### 触发信号

- 感觉“好像对”，但经不起深问。
- 关键前提没有被明说。
- 结论成立依赖若干隐含假设。

#### 典型输出

- 显式假设清单。
- 证据缺口。
- 替代解释。
- 潜在后果。
- 需要补强的问题点。

#### 不适用场景

- 纯信息盘点。
- 简单执行动作。
- 只需要快速收集素材的时候。

#### 一句话

`苏格拉底提问` 是纵向追问，用来补逻辑。

### 4.3 SWOT 分析

#### 核心定义

在项目启动或重大选择前，用优势、劣势、机会、威胁四个象限系统扫描内外部条件，让“能不能做、值不值得做”从感觉判断变成有依据的判断。

#### 解决的问题

- 立项或启动判断依赖感觉拍板。
- 对资源、约束和外部环境看得不全。
- 不确定当前是不是适合推进。

#### 适用场景

- 项目启动前判断。
- 方向筛选。
- 阶段性 go/no-go 决策。
- 多个候选方向的初步比较。

#### 触发信号

- 对“我到底能不能做”存在犹豫。
- 想做，但担心资源或环境不支持。
- 有多个方向，缺少有依据的判断框架。

#### 典型输出

- 四象限判断表。
- 关键优势、劣势、机会、威胁清单。
- 是否推进的结论与理由。

#### 不适用场景

- 已明确执行中的细节问题。
- 纯技术排障。
- 日常小任务处理。

#### 一句话

`SWOT` 不是把问题拆细，而是把“要不要做”看全。

### 4.4 第一性原理

#### 核心定义

当现有路径主要建立在惯例、类比和默认做法上时，把这些外层剥掉，只保留可追溯的基本事实，再从这些事实重新构造长期策略。

#### 解决的问题

- 长期策略被行业惯例和路径依赖绑住。
- 现方案能用，但总觉得不是最好。
- 大家都在按类比走，底层依据不够硬。

#### 适用场景

- 长期策略制定。
- 系统级设计。
- 方法论升级。
- 重大方向判断。
- 路线重构。

#### 触发信号

- 明显在沿用行业惯例。
- 怀疑存在更优路径，但暂时说不清。
- 需要从模仿走向创新。

#### 典型输出

- 底层事实清单。
- 失效惯例清单。
- 重构后的战略假设。
- 替代路径建议。

#### 不适用场景

- 日常执行。
- 小问题修补。
- 已有明确边界下的局部优化。

#### 一句话

`第一性原理` 是把既有策略拆回底层事实。

### 4.5 反向推导

#### 核心定义

当远期目标和当前行动之间存在大段空白时，从终点向前逐层拆解成立所需的前置条件，直到落到当前可以执行的第一个小目标。

#### 解决的问题

- 目标很远，但不知道第一步做什么。
- 中间断层太大，难以下手。
- 大任务无法转成可执行动作。

#### 适用场景

- 路线图设计。
- 长周期项目拆解。
- 复杂目标推进。
- 卡住时寻找下一步动作。

#### 触发信号

- 愿景很清晰，但路径不清晰。
- 目标太大，团队无从开始。
- 中期阶段没有被拆出来。

#### 典型输出

- 终点条件。
- 分层前置条件链。
- 阶段目标。
- 当前最小可执行动作。

#### 不适用场景

- 只是做全维盘点时。
- 只是验证现有结论是否扎实时。

#### 一句话

`反向推导` 是纵向拆解，用来从远目标倒推当前动作。

### 4.6 金字塔原理

#### 核心定义

当议题复杂、观点分散、表达抓不住重点时，先明确塔尖结论，再把支撑论据按层级归拢，让表达从散乱变成可一口讲清。

#### 解决的问题

- 观点很多，但抓不住重点。
- 说不清、写不顺。
- 论据和观点没有层级归属。

#### 适用场景

- 汇报。
- handoff。
- 项目结论表达。
- 知识转写。
- 决策摘要。

#### 触发信号

- 已经有内容，但表达混乱。
- 想讲清楚，却越讲越散。
- 需要把复杂议题压缩成清晰主线。

#### 典型输出

- 一句核心结论。
- 2 到 4 组支撑论据的层级结构。
- 可快速复述的结构化表达。

#### 不适用场景

- 事实还没查清。
- 问题还没定义清楚。
- 仍处在发散探索阶段。

#### 一句话

`金字塔原理` 不是用来想出答案，而是用来把答案讲清楚。

### 4.7 六顶思考帽

#### 核心定义

当一个问题总在单一视角里打转、越想越卡时，强制按不同思考角度依次切换，让每个角度独立产出、互不混杂，从单线思考转成多维覆盖。

#### 解决的问题

- 思路卡死在一个角度。
- 事实、判断、情绪、创意混在一起。
- 多人或多 Agent 讨论时相互打断、互相污染。

#### 适用场景

- 多 Agent 协作讨论。
- 复杂方案评审。
- 有争议的方向判断。
- 创意和风险并存的议题。

#### 触发信号

- 同一个角度反复重复。
- 信息互相打架。
- 讨论越多越乱。

#### 典型输出

- 按视角分栏的独立结论。
- 多视角汇总结果。
- 视角间的冲突点与互补点。

#### 不适用场景

- 简单事实查询。
- 明确执行任务。
- 已经只差落地动作的问题。

#### 一句话

`六顶思考帽` 不是为了把问题拆成步骤，而是为了把视角拆开。

### 4.8 批判性思维

#### 核心定义

当一个重要方案已经形成、准备提交，但还不确定是否存在盲点时，用“前提、证据、逻辑、替代解释”四项检验主动寻找薄弱环节并补强，让方案从“希望别被质疑”升级到“准备好被质疑”。

#### 解决的问题

- 方案已成形，但还不确定是否经得起挑战。
- 存在前提风险、证据不足或逻辑跳步。
- 还没主动考虑更好的解释。

#### 适用场景

- 重要方案提交前。
- handoff 发出前。
- 项目结论写入前。
- 知识定稿前。
- 决策建议输出前。

#### 触发信号

- 内容重要，影响较大。
- 需要经得起质疑和复核。
- 自己隐约感觉可能还有盲点。

#### 典型输出

- 前提风险。
- 证据缺口。
- 逻辑断点。
- 替代解释。
- 对应补强动作。

#### 不适用场景

- 早期发散。
- 纯信息收集。
- 简单执行动作。

#### 一句话

`批判性思维` 不是帮你产出第一个答案，而是帮你把答案打磨到更能扛质疑。

## 7. 当前结构关系

这 8 个模式目前可先按如下关系理解：

- `5W1H`：看全。
- `苏格拉底提问`：追深。
- `SWOT`：判断要不要做。
- `第一性原理`：重构长期策略。
- `反向推导`：拆目标。
- `金字塔原理`：讲清楚。
- `六顶思考帽`：换视角。
- `批判性思维`：提交前总复核。

## 8. 对 OrbitOS 的意义

这套模式库最重要的价值，不是让 Agent 背概念，而是给 OrbitOS 一个稳定的思考方法层：

- 遇到不同任务时，Agent 不再总用同一套默认脑回路。
- 多 Agent 协作时，可以明确这次用了什么思考方法，而不是只靠个人习惯。
- 后续角色模型、Profile、handoff 模板和 validation，都可以引用这些模式，而不是把思考方式散落在经验里。

## 9. 最小触发建议

为避免“知道模式但不会用”，当前先固定一组最小触发建议：

- 任务目标模糊、担心有盲区：优先 `5W1H`
- 远期目标清楚但不知道第一步：优先 `反向推导`
- 启动项目前犹豫能不能做：优先 `SWOT 分析`
- 长期策略明显受惯例绑架：优先 `第一性原理`
- 讨论陷入单一视角：优先 `六顶思考帽`
- 结论像是对的但不够扎实：优先 `苏格拉底提问`
- 已有答案但表达混乱：优先 `金字塔原理`
- 方案准备提交但担心有盲点：优先 `批判性思维`

这些建议默认作为弱触发推荐，不是必须命中的强规则。

## 9.1 薄触发表

只保留一版不超过一屏的推荐触发表：

| 任务信号 | 默认推荐 |
|---|---|
| 目标模糊，不确定缺了什么 | `5W1H` |
| 目标很大，不知道第一步怎么做 | `反向推导` |
| 正在犹豫要不要启动 | `SWOT 分析` |
| 怀疑行业惯例不对，想从头重构 | `第一性原理` |
| 观点很多，但讲不清主线 | `金字塔原理` |
| 一直卡在单一视角打转 | `六顶思考帽` |
| 结论看起来对，但不够扎实 | `苏格拉底提问` |
| 方案要提交，担心还有盲点 | `批判性思维` |

补充约束：

- 默认一次只推荐 `1` 到 `2` 个模式。
- 没有明显任务信号时，不推荐。
- 推荐是提示，不是强制执行合同。
- 若用户已指定模式，覆盖默认推荐。
- 只有当本轮实际采用了思考模式时，才标注 `本轮思考模式`；未采用时不标注。

## 10. 组合边界

以下组合默认合理：

- `5W1H` + `反向推导`
- `反向推导` + `金字塔原理`
- `SWOT 分析` + `批判性思维`
- `六顶思考帽` + `批判性思维`
- `苏格拉底提问` + `金字塔原理`

以下边界需要明确：

- 不要求把 8 个模式在同一任务里全部跑一遍。
- 不要求 handoff 必须把思考过程展开成完整模板。
- 不把任一模式包装成唯一正确路径。
- 不把思考方法层误写成系统已经自动执行的隐藏流程。
