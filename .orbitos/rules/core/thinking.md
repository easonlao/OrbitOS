# Thinking Method Layer

思考方法层是核心的按需认知能力，不是可安装模块、角色、Skill 或额外 workflow。

- 角色回答谁负责什么职责。
- Skills 回答能调用什么能力。
- workflow 回答按什么执行步骤推进。
- 思考方法层回答当前该用什么方法把问题想清楚、讲清楚、校验清楚。

## 调用边界

- 只在目标或范围模糊、存在重要取舍、需要复杂拆解、需要审查验证，或用户明确要求先分析时触发。
- 简单查询、格式修改、已明确步骤的执行和纯命令操作不触发。
- 命中后必须先展示思考模式启动选择；不得静默替用户选定方法，也不得在用户选择前进入实质分析、设计、写入或决策。
- 一次只给 `2` 到 `3` 个候选，并明确推荐项及原因；不要一次抛出全部八种方法。
- 用户可回复数字、方法名称或自然语言意图；`直接做`、`跳过思考模式` 是本轮明确 bypass。
- 只有实际采用时，才在输出和 event 中标注“本轮思考方式”。
- 不把任何模式包装为唯一正确路径，也不伪装成系统自动执行的隐藏流程。

## 思考模式启动选择

命中触发器时，先用以下格式向用户展示选择卡：

```text
这一步需要先确定分析方式：

1. {方法名}
   {一句话特点与当前任务的适配原因}

2. {方法名}
   {一句话特点与当前任务的适配原因}

3. {可选的第三种方法}
   {一句话特点与当前任务的适配原因}

推荐：{编号}，因为 {当前任务的具体原因}。
回复 1 / 2 / 3；也可以说“直接做”或“看全部方法”。
```

规则：

- 用户要求“看全部方法”时，给出八种方法各一行的名称、特点和适用信号，再等待选择。
- 用户以自然语言描述意图时，先映射为最接近的方法，并用一句话确认映射；不要求用户记住名称。
- 用户选定后，只读取 `thinking-modes.md` 中对应章节；组合最多 `2` 种。
- 用户 bypass 后直接推进，但不得在输出或 event 中宣称使用了思考方法。
- 选择只对当前任务有效；除非用户明确要求，不把选择写成长期偏好。

## 选择后的可追溯性

- 实质性工作完成后，输出一行 `本轮思考方式：{方法}。作用：{用途}`。
- 写 completion event 时，使用 `thinking` 字段记录 `selected` 或 `bypassed`、实际方法和简短用途；不记录完整推理过程。

## 详细方法

本文件只负责触发、选择与调用边界。确定采用某种模式后，读取 `.orbitos/rules/core/thinking-modes.md` 中该模式的对应章节；不要为简单任务加载整份详细词典。

- `5W1H`：读取 `### 4.1 5W1H`
- `苏格拉底提问`：读取 `### 4.2 苏格拉底提问`
- `SWOT 分析`：读取 `### 4.3 SWOT 分析`
- `第一性原理`：读取 `### 4.4 第一性原理`
- `反向推导`：读取 `### 4.5 反向推导`
- `金字塔原理`：读取 `### 4.6 金字塔原理`
- `六顶思考帽`：读取 `### 4.7 六顶思考帽`
- `批判性思维`：读取 `### 4.8 批判性思维`

## 接入面

- 分析：澄清、拆解、重构与判断。
- handoff：复杂交接时说明实际采用的模式。
- review / 自检：提交前整理表达、补逻辑与最终复核。

## 最小触发

| 任务信号 | 默认推荐 |
|---|---|
| 目标模糊，不确定缺了什么 | `5W1H` |
| 目标很大，不知道第一步怎么做 | `反向推导` |
| 正在犹豫要不要启动 | `SWOT 分析` |
| 怀疑行业惯例不对，想从头重构 | `第一性原理` |
| 观点很多，但讲不清主线 | `金字塔原理` |
| 一直卡在单一视角打转 | `六顶思考帽` |
| 结论看起来对，但不够扎实 | `苏格拉底提问` |
| 方案要提交，担心还有盲点 | `批判性思维` |

合理组合保持克制，例如 `5W1H + 反向推导`、`反向推导 + 金字塔原理`、`SWOT 分析 + 批判性思维`。不要求同一任务跑完全部模式。
