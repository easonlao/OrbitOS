# Knowledge Conflict Workflow

Knowledge Conflict 用于处理新证据与现有 active knowledge 核心结论冲突的情况。

## 目标

- 当两份已提炼知识在核心结论上冲突时，不让任一版本继续直接占据当前知识位置。
- 将冲突中的两份知识版本都退回到 `01-收件箱/已入库/`，作为后续重整时的索引源。
- 在 `04-知识/00-草稿箱/` 新建一份“冲突内容详解”草稿，供用户判断最终只输出哪一份当前知识。

## 适用范围

适用于以下情况：

- 两份已提炼知识在核心结论上冲突。
- 新证据会改变现有 active knowledge 的核心结论，并已经形成了另一份可独立表达的知识版本。
- 你希望把冲突重新收敛为一份新的当前知识，而不是长期并列保留两个 active 卡片。

它不是首次知识提炼。区别是：

- `knowledge-draft.md`：原始输入第一次转写为知识草稿。
- `knowledge-conflict.md`：两份冲突知识先退出知识区，回到已入库索引源，再生成一份冲突详解草稿供用户确认最终输出。

## 输入

- 至少两份在结论上冲突的知识版本。
- 支撑这些知识版本的已入库原始输入。
- 可选的补充上下文或相关知识笔记。

## 输出

输出草稿放在：

```text
04-知识/00-草稿箱/
```

frontmatter 使用可见 Markdown 最小字段：

```yaml
---
title:
area: knowledge
purpose: knowledge
lifecycle: draft
created:
updated:
tags:
---
```

正文必须包含：

- 两份冲突知识版本的来源链接
- 两边各自的核心结论
- 冲突点
- 证据差异
- 建议处理方式

## 执行流程

1. 确认用户允许处理这次知识冲突。
2. 读取两份冲突知识版本和它们对应的已入库原始输入。
3. 读取 `.orbitos/rules/core/markdown-writing.md`。
4. 判断冲突是否真的涉及核心结论；如果只是补充来源且不改变语义，不使用本 workflow。
5. 将冲突中的两份知识版本退出知识区，并转存到 `01-收件箱/已入库/` 作为索引源。
6. 在 `04-知识/00-草稿箱/` 新建一份“冲突内容详解”草稿。
7. 在草稿中写明：
   - 版本 A 的核心结论
   - 版本 B 的核心结论
   - 两者具体冲突在哪里
   - 各自依赖的证据是什么
   - 建议处理方式：保留 A / 保留 B / 合并重写为一份新知识
8. 不修改、覆盖、重命名或移动 `01-收件箱/已入库/` 中的原始输入。
9. 把待确认冲突草稿投影到 `今日.md`。
10. 执行 Validate Sync。
11. 执行 Progress Sync。

## 用户确认后的处理

用户确认后，根据结论执行：

1. 保留 A：
   - 基于冲突草稿重新输出一份新的当前知识，结论以 A 为准。
2. 保留 B：
   - 基于冲突草稿重新输出一份新的当前知识，结论以 B 为准。
3. 合并重写：
   - 基于冲突草稿重新输出一份新的当前知识，在一张卡片中重写边界、条件和最终判断。

无论走哪条路径，最终进入知识区的都应只有一份新的当前知识。

## 异常分支

- 冲突版本还没有可追溯来源：停止并先补来源。
- 冲突其实只是措辞差异、不改变结论：不走冲突 workflow，按普通维护处理。
- 用户并不想重新收敛为一份知识：停止并请用户明确是否真的属于知识冲突。

## 执行清单

### 进入检查

- [ ] 已确认当前 workflow 适用。
- [ ] 已确认用户允许处理这次知识冲突。
- [ ] 已确认存在两份冲突知识版本及其来源。
- [ ] 已读取 Markdown 写作规则。

### 执行检查

- [ ] 已将两份冲突知识版本转存为已入库索引源。
- [ ] 已创建冲突内容详解草稿。
- [ ] 已写明两边结论、冲突点和处理路径。
- [ ] 已保留知识版本与原始输入的可追溯链接。
- [ ] 已把待确认冲突草稿投影到用户可见入口。

### 退出检查

- [ ] 已执行 Validate Sync 或 validation eval。
- [ ] 已执行 Progress Sync 并写入 event。
- [ ] 已记录所有跳过、失败或阻塞项和原因。

## 禁止

- 不因新证据出现就直接改写当前知识。
- 不把冲突中的知识版本继续留在知识区并假装两者都还是当前知识。
- 不把原始输入当成最终知识。
- 不在用户确认前决定最终输出哪一份当前知识。
