# Obsidian + AI Agent 实践案例

Eric J Ma（2026.03）分享了他用 Obsidian + AI coding agents 管理工作的完整方案。管理 12 人、跨两个团队、每人 2-4 个项目，知识管理开销从 30-40% 降到 10% 以下。

## 核心决策：纯文本

选择 Obsidian 而非 Confluence/OneNote 的关键原因：
- **无供应商锁定**：文本文件是最原始的格式
- **AI 原生可读**：AI agents 到来时，vault 已经是可处理的格式
- **零迁移成本**：不需要转换层或 API 集成

这个决策在 2022 年做出，到 2026 年被证明极具前瞻性。纯文本恰好是 AI 时代知识管理的最佳格式。

## Vault 结构

| 笔记类型 | 用途 |
|---------|------|
| 月度日志 | 每月一个文件，记录每日活动 |
| 会议笔记 | 结构化模板，含转录/AI 摘要 |
| 人物档案 | 每个合作对象的 dossier |
| 项目笔记 | 控制塔，链接会议、人物、状态 |
| 杂项 | 其他一切 |

关键：**AGENTS.md 文件让 AI agents 理解整个系统**。

## 信息摄入流程

### 会议记录
- 输入：AI 生成摘要、有/无说话人标注的转录
- 处理：Meeting notes skill 提取关键信息，统一格式
- 输出：同时挂到会议日志和人物页面

### 文档处理
- **Word**：python-docx 转纯文本 → agent 读取
- **PPT**：双路径——XML 结构提取 + 逐页图片转文字（VLM captioning），90-95% 准确率
- **PDF**：文本提取（正常）/ 图片 captioning（扫描件）
- **Excel**：openpyxl 读取细胞级结构，不假设表格结构

关键洞察：**PPT 用 openpyxl 而非 pandas**，因为真实 Excel 很混乱，需要先识别架构再深入内容。

## 维护机制

### Sweep（扫掠）
触发条件：查看某人/某项目时发现信息缺失
流程：
1. Agent 扫描 vault 中的源材料
2. 更新人物/项目笔记
3. 必须引用原始来源的引文
4. 人工验证

### 检索练习
定期检查人物/项目笔记：
- 有什么记忆中的信息没被记录？
- 已记录的声明是否有链接和引文支撑？

## 对 OrbitOS 的启示

1. **AGENTS.md 是系统的灵魂**——让 agent 理解 vault 结构和工作流
2. **结构化笔记类型**比扁平存储更有用
3. **Sweep 机制**可以自动化：定期扫描收件箱，更新关联笔记
4. **纯文本 + AI**的组合是经过验证的最佳实践
5. **质量控制**需要人工参与，agent 做初筛，人做验证

## 局限与边界

- 人物档案是专业性质的，不涉及个人隐私
- Hallucination 率约 1/5 次扫掠，通常源于不准确的转录而非 agent 错误
- 系统需要持续维护，不是"设置好就忘"
