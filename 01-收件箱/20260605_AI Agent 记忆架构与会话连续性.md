# AI Agent 记忆架构与会话连续性

## 核心问题

AI session 是无状态的。每次对话从零开始。如果 vault 只是一堆笔记的集合，没有专门的"面向 AI 的结构层"，每次会话要么靠冗长的 briefing 开场，要么在错误假设中浪费 token。

## 方案一：三层记忆架构（Moun R.）

将记忆分为三层，各司其职：

| 层级 | 内容 | 读取频率 | 生命周期 |
|------|------|----------|----------|
| **L1 工作记忆** | 原始会话日志，带时间戳 | 不主动读取 | 短期，原材料 |
| **L2 策展记忆** | MEMORY.md，蒸馏后的关键信息 | 每次主会话读取 | 中期，持续更新 |
| **L3 持久知识** | Obsidian vault 中的指南、画像、知识库 | 按需读取 | 长期，永久保存 |

### 信息流动路径

```
会话发生 → L1 写入原始日志
         → L2 心跳周期蒸馏（每天 2-4 次安静期）
         → L3 沉淀为持久知识
```

### 关键设计原则

- **策展 > 堆积**：MEMORY.md 只保留 agent 无法从其他来源重建的信息（用户偏好、当前优先级、近期教训），不复制配置值或已有的规则
- **简洁 > 详尽**：从 1200 词压缩到 548 词，行为不变，剩余 token 留给实际对话
- **读 ≠ 记住**：模型注意力有限，长文件会被扫读，短而结构化的文件才能被有效吸收
- **agent 自己做蒸馏**：心跳周期中读取近期日志 → 识别值得保留的 → 更新 MEMORY.md，不是备份系统，是反思循环

## 方案二：面向 AI 的 Vault 结构化（andrefoti）

从 vault 结构层面解决无状态问题，三个组件：

### 1. 上下文文件（Context Files）

```
Working Context/
├── OrbitOS.md          ← 当前状态、开放问题、相关文件引用
├── 研究方向.md         ← 活跃研究领域的当前进展
└── ...
```

每个活跃项目/领域一个文件，头部写清：是什么、当前状态、待解决问题、相关引用。关键纪律：**在会话结束时更新，不是事后想起来才补**。

### 2. 会话启动协议（Session Start Protocol）

每次会话开始前固定读取：
1. vault 级导航文件（全局结构概览）
2. 规范动作寄存器（当前待办事项）
3. 如有领域聚焦，读取对应上下文文件

编码为可复用的 skill，不依赖 AI 自行推断该读什么。

### 3. 交接协议（Handoff Protocol）

会话结束时生成结构化文件：

```markdown
## 完成事项
- ...

## 需要在 Obsidian 中人工审核
- ...

## 推迟事项
- ...

## 下一步
- ...
```

下一次会话打开此文件，弥合无状态的断裂感。

## 方案三：obsidian-memory-for-ai（SPEC-v3）

将 AI 记忆定义为一组 Markdown + YAML 文件的协议规范：
- 纯文本、可读可编辑可 diff 可同步
- 人和 agent 共用同一套文件，但不混杂
- 设计为跨工具可移植

## 三种方案的共同规律

| 规律 | 含义 |
|------|------|
| **Vault 是持久状态层** | Agent session 是无状态 worker，vault 承担记忆职责 |
| **分层比扁平好** | 原始日志、策展记忆、持久知识各有位置 |
| **启动协议必须显式编码** | 不能让 AI "猜"该读什么，用 skill 固化启动流程 |
| **交接弥合断裂** | 每次会话结束时写交接文件，下次会话读取 |
| **策展靠 agent** | 让 agent 自己做蒸馏，人做周期性审核纠偏 |
| **简洁是性能** | 记忆文件越精简，有效注意力窗口越大 |

## 对 OrbitOS 的启示

1. **Nova 的巡检本身就是 L1 工作记忆**：每次巡检日志是时间戳化的会话记录
2. **AGENTS.md 承担了部分 L2 角色**：定义了 agent 行为规则，但缺少"策展记忆层"
3. **缺少显式交接协议**：Nova 在 patrol 之间没有结构化的上下文传递机制
4. **Daily Note 可作为会话锚点**：`10_日记/YYYY-MM-DD.md` 是天然的 L1 日志 + 交接载体
5. **建议增加 Working Context 层**：在 `99_系统/` 或 `OH-Works/` 下维护 Nova 的当前工作上下文文件
