# Git多端同步与群晖NAS最佳实践

针对当前“Windows（本地 Git + GitHub）+ Synology Drive（局域网同步）+ Ubuntu .33（SMB挂载）”的混合开发场景，以及“**开源分享系统框架，但局域网私密同步笔记内容**”的特定需求，特制定此双轨同步工作流规范。

---

## 1. 局域网设备 IP 与角色映射

| 设备名称 | 局域网 IP | 物理位置/路径 | 同步角色 |
| :--- | :--- | :--- | :--- |
| **Windows PC** | `10.10.10.70` | 本地 E 盘 (`e:\SynologyDrive\OrbitOS`) | **Git 提交中心**，唯一保留本地 `.git` 文件夹的设备 |
| **Synology NAS** | `10.10.10.30` | 共享文件夹 (SMB 服务) | **内容存储中枢**，通过 Drive 接收 Win 同步，提供 SMB 给 Ubuntu |
| **Ubuntu VM/PC** | `10.10.10.33` | SMB 挂载目录 (`~/orbit`) | **远程执行节点**，不包含 `.git` 文件夹，仅通过 SMB 读写代码/数据 |

---

## 2. 双轨同步架构设计

为同时满足“系统框架开源分享”与“个人笔记隐私同步”两个诉求，系统采用**“Git 跟踪框架 + Synology Drive 同步全部（排除 `.git`）”**的隔离设计：

```
                      【GitHub】 (仅包含系统框架与规则)
                          ▲
                          │ git push / pull (仅在 Windows 10.10.10.70 执行)
                          ▼
      [Windows 本地 (10.10.10.70)] (独占本地 .git 目录)
            │
            │ Synology Drive 实时同步 (排除 .git 文件夹)
            ▼
      [NAS / SMB 共享 (10.10.10.30)] (无 .git 文件夹，仅有代码 and 笔记)
            ▲
            │ SMB 实时读写 (挂载至 ~/orbit)
            ▼
       [Ubuntu (10.10.10.33)] (无 .git 文件夹，仅有代码 and 笔记)
```

### 1) 个人内容（笔记）的流向：
* **实时同步**：你在 Windows (`10.10.10.70`) 或 Ubuntu (`10.10.10.33`) 写的个人笔记，会通过 Synology Drive 在 NAS (`10.10.10.30`) 间进行局域网同步。
* **安全性**：由于 `.gitignore` 的保护，你的个人私密笔记（`01-收件箱`、`02-日记`、`03-知识` 等）**绝对不会**被上传到 GitHub。

### 2) 框架系统文件（代码/规则）的流向：
* **修改**：你可以在两端任意修改系统配置或规则，改动均会通过局域网自动同步。
* **提交与分享**：当你需要发布框架版本时，**统一在 Windows 本地执行 Git 提交**，然后推送到 GitHub。

---

## 3. 为什么不能在两端同步 `.git` 文件夹？

1. **仓库损坏风险 (Corruption)**：Synology Drive 是文件级异步同步工具。在执行 Git 写操作时，若同步软件进行读取锁定或产生单向同步覆盖，极易导致 `.git` 数据库损坏（如 `loose object corrupt` 报错）。
2. **SMB 性能灾难**：如果 Ubuntu (`10.10.10.33`) 直接在挂载的 SMB 目录下运行 Git 命令（或 IDE 的 Git 插件扫描），网络延迟会导致 `git status` 极其缓慢甚至引起编辑器卡死。
3. **两端状态分裂**：Windows 和 Linux 的文件系统差异，会导致同一个同步 `.git` 在两端交替报错。

## 4. 具体配置与清理指南

### 步骤一：确保 `.gitignore` 隔离隐私内容
在 Windows 端的 `.gitignore` 中，已确保将所有日常内容目录下的 `.md` 文件排除，仅保留结构模板文件（`WORKSPACE.md`）：
```gitignores
# ── 日常内容文件（模板 repo 不包含个人数据） ────────────────────
02-日记/工作日志/*.md
02-日记/反思/*.md
01-收件箱/**/*.md
03-知识/**/*.md
04-项目/**/*.md
...
!01-收件箱/WORKSPACE.md
!03-知识/WORKSPACE.md
...
```

### 步骤二：在 Synology Drive 中排除 `.git`
1. 打开 Windows `10.10.10.70` 上的 **Synology Drive Client**。
2. 进入 **同步任务** -> 选择当前任务 -> 点击 **同步规则**。
3. 切换到 **文件过滤器**，确保勾选了 `.*` 文件夹或手动添加 `.git` 文件夹排除。

### 步骤三：清理 Ubuntu/NAS 端的残留 `.git`
由于此前 `.git` 已经被同步到了 NAS，请在 **Ubuntu (`10.10.10.33`)** 上手动将 SMB 挂载目录下的旧 `.git` 目录删除：
```bash
# 在 Ubuntu 上执行，清除 NAS 残留
cd ~/orbit
rm -rf .git
```
*（执行后，在 Ubuntu 运行 `git status` 应当提示 `fatal: not a git repository`，说明清理完成）*

---

## 5. 日常开发与提交工作流

1. **开发与写作**：
   * 在 Windows (`10.10.10.70`) 或 Ubuntu (`10.10.10.33`) 上正常开发或记笔记。
2. **框架代码提交**：
   * **只在 Windows 端操作**。当你在 Ubuntu 上修改了框架文件，回到 Windows 电脑：
     1. 确保任务栏 Synology Drive 客户端显示 **“已是最新”**（这确保了 Ubuntu 的修改已从 `10.10.10.30` 同步回 Windows 本地）。
     2. 运行 `git add`、`git commit` 提交。
     3. 运行 `git push origin main` 推送到 GitHub。
